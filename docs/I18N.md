# UI translation (controlled i18n)

The app uses small JSON locale files and a shared [`i18n.js`](../i18n.js) module—no full-page machine translation, so the product name **Bingabu** and join flow stay predictable.

## Runtime delivery

- Browsers call **`GET /api/locales/{en|he}`** first (merged: static `locales/*.json` + Redis overrides). If that fails, they fall back to static **`/locales/*.json`** files.
- **Redis overrides** are optional in local static-only dev; on Vercel, use the same Upstash credentials as rooms.

## Admin UI ([`admin-translations.html`](../admin-translations.html))

1. Set **`BINGABU_ADMIN_LOCALES_TOKEN`** in the deployment environment to a long random secret (never commit it).
2. Ensure **`UPSTASH_REDIS_REST_URL`** and **`UPSTASH_REDIS_REST_TOKEN`** are set—**saving** from the admin UI writes override blobs to Redis (`bingabu:i18n:overrides:en`, `:he`). Without Redis, you can still **load** the editor but **Save** returns 503.
3. Open **`/admin-translations.html`**, paste the token, click **Connect**, edit the table, **Save all changes**.
4. The page is **`noindex`**; the URL is not linked from the public game UI—share it only with people who may edit copy.

**Workflow:** Git remains the baseline for **new keys** and code review. The admin UI is for **tweaking live strings** without a deploy. After large edits, consider copying values back into `locales/*.json` in git so defaults stay in sync.

Protected API: **`GET`** and **`PUT /api/admin-locales`** with header `Authorization: Bearer <token>`.

## Adding a string

1. Add the English value to [`locales/en.json`](../locales/en.json) under a stable dotted key (e.g. `join.error.enterCode`).
2. Add the same key to [`locales/he.json`](../locales/he.json) (and any future locale files). **Keep “Bingabu” in Latin** in Hebrew strings where it names the product.
3. **Static HTML:** put `data-i18n="your.key"` on the element. For `placeholder`, `aria-label`, or `title`, add `data-i18n-attr="placeholder"` (space-separated for multiple attributes). Elements with only attributes and no visible text (e.g. `<input>`) use `data-i18n-attr` only; containers that must not lose children (e.g. overlay `<div>`) use `data-i18n-attr="aria-label"` only.
4. **JavaScript:** `import { initI18n, t } from "./i18n.js"`, `await initI18n()` once at startup, then `t("your.key")` or `t("join.slot.playerN", { n: 3 })` for `{n}` placeholders.

## Adding a language

1. Add the locale code to `SUPPORTED_LOCALES` in [`i18n.js`](../i18n.js).
2. Copy `en.json` to `locales/<code>.json` and translate.
3. If the language is RTL, set `dir` in `initI18n` (see Hebrew: `document.documentElement.dir = locale === "he" ? "rtl" : "ltr"`).
4. Add a **head inline script** (see `index.html`) that mirrors locale resolution so `lang`/`dir` are set before first paint.
5. Add font overrides under `html[lang="…"]` if the default stack lacks glyphs.

## Locale resolution order

1. `?lang=` query parameter (also persisted to `localStorage` under `bingabu-locale`)
2. `localStorage` (`bingabu-locale`)
3. `navigator.language` (prefix match, e.g. `he-IL` → `he`)
4. Default `en`
